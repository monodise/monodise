ПМ 9 - Use Case
Билеты 1-5. Полные ответы для подготовки к экзамену на 90-100 баллов.
Краткая памятка
Билет №1. Основы Use Case диаграммы
1. Что такое Use Case диаграмма? Назовите ее основные элементы и их назначение.
Use Case диаграмма, или диаграмма вариантов использования, - это UML-диаграмма, которая применяется для описания функциональных требований к системе с точки зрения внешнего пользователя. Она показывает, кто взаимодействует с системой и какие функции система предоставляет этим пользователям.
Главная задача Use Case диаграммы - показать, что должна делать система, не углубляясь во внутреннюю реализацию. На такой диаграмме не показывают программный код, таблицы базы данных или алгоритмы. Она описывает цели пользователей и функции системы.
Основные элементы Use Case диаграммы:
- Актор - внешняя роль, взаимодействующая с системой.
- Вариант использования - функция системы, изображаемая овалом.
- Граница системы - прямоугольник, который показывает область ответственности системы.
- Связи - линии взаимодействия между акторами и вариантами использования, а также связи Include, Extend и Generalization.
Use Case диаграммы полезны на этапе анализа требований. Они помогают заказчику, преподавателю и разработчику одинаково понимать, какие функции должна иметь будущая программа.
2. Что такое актор? Чем первичный актор отличается от вторичного?
Актор - это внешняя сущность, которая взаимодействует с системой. Важно понимать, что актор - это роль, а не обязательно конкретный человек. Один человек может выступать в разных ролях, например как покупатель и администратор.
Первичный актор инициирует выполнение сценария и получает основной результат. Вторичный актор помогает системе выполнить сценарий, предоставляет данные или внешний сервис.
Например, в интернет-магазине покупатель является первичным актором, потому что он оформляет заказ. Платежная система является вторичным актором, потому что она помогает провести оплату.

Билет №2. Include, Extend и Generalization
1. Что такое Include и Extend? Чем они отличаются?
Include и Extend - это связи между вариантами использования. Они применяются для уточнения сценариев и уменьшения повторений на диаграмме.
Include означает обязательное включение одного варианта использования в другой. Если основной сценарий выполняется, включаемый сценарий выполняется всегда. Например, "Оформление заказа" обязательно включает "Авторизацию", если система требует входа в аккаунт перед заказом.
Extend означает необязательное расширение сценария. Расширяющий Use Case выполняется только при определенном условии. Например, "Применить промокод" расширяет "Оформление заказа", если у покупателя есть промокод.
Главное отличие: Include выполняется всегда, а Extend выполняется только при условии. В Include стрелка идет от основного варианта к включаемому. В Extend стрелка идет от расширяющего варианта к основному.
2. Что такое Generalization?
Generalization - это отношение обобщения, то есть наследования. Оно может использоваться между акторами или между вариантами использования. Дочерний элемент наследует возможности родительского и может иметь собственные дополнительные особенности.
Пример между акторами: общий актор "Пользователь", а дочерние акторы - "Покупатель" и "Администратор". Пример между Use Case: общий вариант "Оплатить заказ", а частные варианты - "Оплатить картой" и "Оплатить наличными".

Билет №3. Документирование Use Case
1. Как документируется вариант использования?
Для подробного описания варианта использования применяется текстовый шаблон Use Case. Диаграмма показывает функцию кратко, а шаблон объясняет сценарий работы системы шаг за шагом.
- Название - краткое имя сценария.
- Актор - роль, запускающая сценарий.
- Предусловие - что должно быть выполнено до начала.
- Основной поток - нормальный сценарий выполнения.
- Альтернативный поток - ошибки, исключения или другие варианты.
- Постусловие - результат после завершения сценария.
Такой шаблон помогает разработчику, заказчику и преподавателю одинаково понимать требования к системе.
2. Как Use Case связан с другими UML-диаграммами?
Use Case диаграмма является отправной точкой проектирования. Она показывает, что должна делать система. После этого отдельные варианты использования можно детализировать другими UML-диаграммами.
Диаграмма активности показывает поток действий внутри сценария. Диаграмма последовательностей показывает взаимодействие объектов во времени. Диаграмма классов показывает структуру системы, классы, атрибуты и связи. Например, из Use Case "Оформление заказа" можно выделить классы "Пользователь", "Заказ", "Товар", "Корзина".
3. Практическое задание. Описание Use Case "Оформление заказа"
Основной поток:
- Покупатель открывает корзину.
- Система показывает товары, количество и сумму.
- Покупатель проверяет заказ.
- Покупатель вводит адрес доставки.
- Покупатель выбирает способ оплаты.
- Покупатель подтверждает заказ.
- Система создает заказ и присваивает ему номер.
Альтернативный поток: если товар отсутствует на складе, система сообщает об этом и предлагает изменить корзину или удалить отсутствующий товар.
Билет №4. Граница системы и определение Use Case
1. Что такое граница системы?
Граница системы - это прямоугольник на Use Case диаграмме, внутри которого находятся варианты использования. Она показывает область ответственности системы. Акторы располагаются снаружи, потому что они являются внешними участниками.
Граница системы помогает понять, какие функции выполняет сама программа, а какие действия выполняют внешние пользователи или сервисы. Без границы системы диаграмма может стать неоднозначной.
2. Как определить варианты использования?
Первый подход - от актора: определяют всех акторов и для каждого задают вопрос, какие цели он достигает с помощью системы. Второй подход - от бизнес-процессов: анализируют процессы организации и выделяют функции, которые должна автоматизировать система.
Use Case должен быть не слишком мелким и не слишком общим. Например, "Нажать кнопку" - слишком мелко, а "Работать с системой" - слишком общо. Хороший Use Case описывает полезный результат для актора.

Билет №5. Use Case и диаграмма активности
1. Разница между Use Case и диаграммой активности
Use Case диаграмма показывает, что система делает для пользователя. Диаграмма активности показывает, как выполняется процесс внутри системы. Use Case применяют при сборе и анализе требований, а диаграмму активности - при детализации сценариев и алгоритмов.
Например, Use Case "Оформление заказа" показывает саму функцию. Диаграмма активности для этого сценария показывает шаги: открыть корзину, проверить товары, выбрать доставку, оплатить, получить подтверждение.
2. Ошибки при создании Use Case диаграмм
- Слишком много технических деталей реализации.
- Отсутствие акторов.
- Неправильное использование Include и Extend.
- Слишком общие варианты использования.
- Слишком мелкие варианты использования вроде "нажать кнопку".
- Названия не с глагола, а существительными.
Чтобы избежать ошибок, нужно описывать цели пользователя, проверять границу системы, правильно размещать акторов и применять Include/Extend только по смыслу.

ПМ 9 - SQL и базы данных
Билеты 6-10. Полные ответы, ER-модели, SQL-запросы и практические задания.
Билет №6. ER-диаграммы и нормализация
1. Что такое ER-диаграмма? Назовите основные элементы.
ER-диаграмма, или диаграмма "сущность-связь", - это визуальная модель структуры базы данных. Она показывает, какие объекты предметной области будут храниться в базе, какие у них есть атрибуты и как они связаны между собой.
Основные элементы ER-диаграммы: сущность, атрибут, связь и кардинальность. Сущность - объект реального мира, например клиент, товар или заказ. Атрибут - характеристика сущности, например имя клиента, цена товара или дата заказа. Связь показывает взаимодействие между сущностями. Кардинальность определяет количество экземпляров, которые могут участвовать в связи: 1:1, 1:N, N:M.
2. Что такое нормализация? Объясните 1НФ, 2НФ, 3НФ.
Нормализация базы данных - это процесс организации таблиц так, чтобы уменьшить избыточность данных, избежать аномалий обновления и сделать структуру базы более логичной.
- 1НФ: все значения атомарны, нет повторяющихся групп, каждая строка имеет первичный ключ.
- 2НФ: таблица находится в 1НФ, и каждый неключевой атрибут полностью зависит от всего первичного ключа.
- 3НФ: таблица находится во 2НФ, и неключевые атрибуты не зависят друг от друга транзитивно.
Пример: если в таблице заказов хранить имя клиента и телефон клиента, эти данные будут повторяться. Правильнее вынести клиентов в отдельную таблицу clients, а в orders хранить client_id.
3. Практическое задание. База данных учета заказов
CREATE DATABASE IF NOT EXISTS order_system;
USE order_system;

CREATE TABLE clients (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    phone VARCHAR(20),
    email VARCHAR(100)
);

CREATE TABLE products (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    price DECIMAL(10,2) NOT NULL,
    stock INT NOT NULL DEFAULT 0
);

CREATE TABLE orders (
    id INT PRIMARY KEY AUTO_INCREMENT,
    client_id INT NOT NULL,
    date DATE NOT NULL,
    status ENUM('new', 'active', 'completed', 'cancelled') NOT NULL DEFAULT 'new',
    FOREIGN KEY (client_id) REFERENCES clients(id)
);

CREATE TABLE order_items (
    id INT PRIMARY KEY AUTO_INCREMENT,
    order_id INT NOT NULL,
    product_id INT NOT NULL,
    quantity INT NOT NULL,
    price DECIMAL(10,2) NOT NULL,
    FOREIGN KEY (order_id) REFERENCES orders(id),
    FOREIGN KEY (product_id) REFERENCES products(id)
);
В этой структуре данные не дублируются: клиенты, товары, заказы и позиции заказа хранятся отдельно. Связи реализованы через внешние ключи.
Билет №7. JOIN и индексы
1. Что такое JOIN? Виды JOIN и отличие.
JOIN - это операция SQL, которая объединяет строки из двух или нескольких таблиц по заданному условию. Обычно условие записывается после ON и связывает внешний ключ одной таблицы с первичным ключом другой.
- INNER JOIN возвращает только строки, где есть совпадение в обеих таблицах.
- LEFT JOIN возвращает все строки из левой таблицы и совпадения из правой. Если совпадения нет, справа будут NULL.
- RIGHT JOIN возвращает все строки из правой таблицы и совпадения из левой.
2. Что такое индексы?
Индекс - это вспомогательная структура базы данных, которая ускоряет поиск, сортировку и соединение таблиц. Индексы создают на полях, которые часто используются в WHERE, JOIN ON и ORDER BY.
Обычный INDEX ускоряет поиск, а UNIQUE INDEX дополнительно запрещает повторяющиеся значения. Цена индексов - дополнительное место на диске и замедление INSERT, UPDATE и DELETE, потому что индекс тоже нужно обновлять.
CREATE INDEX idx_orders_client_id ON orders(client_id);
CREATE UNIQUE INDEX idx_clients_email ON clients(email);

Билет №8. Транзакции, ACID и процедуры
1. Что такое транзакции? ACID-свойства.
Транзакция - это последовательность операций базы данных, которая выполняется как единое целое. Если все операции успешны, изменения фиксируются командой COMMIT. Если произошла ошибка, изменения отменяются командой ROLLBACK.
- Atomicity - атомарность: транзакция выполняется полностью или не выполняется вообще.
- Consistency - согласованность: база переходит из одного корректного состояния в другое.
- Isolation - изолированность: параллельные транзакции не должны мешать друг другу.
- Durability - долговечность: после COMMIT изменения сохраняются даже при сбое.
START TRANSACTION;
UPDATE products SET stock = stock - 1 WHERE id = 1;
INSERT INTO orders (client_id, date, status) VALUES (1, CURDATE(), 'active');
COMMIT;
2. Что такое хранимые процедуры?
Хранимая процедура - это именованный блок SQL-кода, который сохраняется на сервере базы данных и вызывается по имени. Процедуры применяются для повторного использования кода, централизации логики, снижения сетевого трафика и выполнения сложных операций.
Процедуры могут принимать параметры IN, OUT и INOUT. IN передает значение в процедуру, OUT возвращает значение, INOUT выполняет обе роли.
3. Практическое задание. Процедура GetClientOrders
DELIMITER //

CREATE PROCEDURE GetClientOrders(IN p_client_id INT)
BEGIN
    SELECT o.id AS order_id,
           o.date,
           o.status,
           p.name AS product_name,
           oi.quantity,
           oi.price,
           oi.quantity * oi.price AS item_sum
    FROM orders o
    JOIN order_items oi ON o.id = oi.order_id
    JOIN products p ON oi.product_id = p.id
    WHERE o.client_id = p_client_id
    ORDER BY o.date DESC;
END //

DELIMITER ;

CALL GetClientOrders(1);
Билет №9. VIEW и резервное копирование
1. Что такое VIEW? Чем оно отличается от таблицы?
VIEW, или представление, - это именованный сохраненный SELECT-запрос, который выглядит как виртуальная таблица. Обычно VIEW не хранит данные физически, а показывает результат запроса к реальным таблицам.
Отличие от таблицы: таблица хранит данные, а VIEW хранит логику выборки. Представления используют для упрощения сложных запросов, ограничения доступа к данным и повторного использования SQL-логики.
2. Что такое резервное копирование?
Резервное копирование - это создание копии базы данных для восстановления после ошибки, сбоя, удаления данных или повреждения сервера.
- Полное копирование сохраняет всю базу данных целиком.
- Инкрементальное копирование сохраняет изменения с момента последнего бэкапа.
- Дифференциальное копирование сохраняет изменения с момента последнего полного бэкапа.
mysqldump -u root -p order_system > order_system_backup.sql
3. Практическое задание. VIEW active_orders
CREATE VIEW active_orders AS
SELECT o.id AS order_id,
       c.name AS client_name,
       SUM(oi.quantity * oi.price) AS total,
       COUNT(oi.id) AS item_count
FROM orders o
JOIN clients c ON o.client_id = c.id
JOIN order_items oi ON o.id = oi.order_id
WHERE o.status = 'active'
GROUP BY o.id, c.name;

SELECT *
FROM active_orders
WHERE total > 10000;
Билет №10. NoSQL, ORM и Task Manager
1. Что такое NoSQL базы данных?
NoSQL - это класс баз данных, которые не используют строгую табличную реляционную модель. Они могут хранить данные в виде документов, пар ключ-значение, графов или колонок. NoSQL часто применяют для больших объемов данных, гибкой структуры и горизонтального масштабирования.
Примеры: MongoDB - документная база данных, Redis - ключ-значение, Cassandra - колоночная база данных.
2. Что такое ORM?
ORM, или Object-Relational Mapping, - это технология, которая связывает объекты языка программирования с таблицами базы данных. Разработчик работает с объектами, а ORM формирует SQL-запросы автоматически.
Преимущества ORM: меньше ручного SQL-кода, удобная работа с данными через объекты, защита от SQL-инъекций при правильном использовании, независимость от конкретной СУБД. Примеры ORM: Hibernate, Eloquent, Entity Framework.
3. Практическое задание. Task Manager
Связи: один пользователь имеет много проектов; один проект имеет много задач; одна задача имеет много комментариев; пользователь может оставлять много комментариев.
CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL
);

CREATE TABLE projects (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL,
    name VARCHAR(100) NOT NULL,
    description TEXT,
    FOREIGN KEY (user_id) REFERENCES users(id)
);

CREATE TABLE tasks (
    id INT PRIMARY KEY AUTO_INCREMENT,
    project_id INT NOT NULL,
    title VARCHAR(150) NOT NULL,
    status ENUM('new','in_progress','done') DEFAULT 'new',
    deadline DATE,
    FOREIGN KEY (project_id) REFERENCES projects(id)
);

CREATE TABLE comments (
    id INT PRIMARY KEY AUTO_INCREMENT,
    task_id INT NOT NULL,
    user_id INT NOT NULL,
    text TEXT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (task_id) REFERENCES tasks(id),
    FOREIGN KEY (user_id) REFERENCES users(id)
);

Билет №11. Платформа, режимы, конфигурация и информационная база
1. Что такое 1С:Предприятие? Два режима работы.
1С:Предприятие - это программная платформа для создания и использования бизнес-приложений. На ней разрабатывают системы учета, торговли, склада, зарплаты, бухгалтерии и других бизнес-процессов.
Основные режимы: Конфигуратор и 1С:Предприятие. Конфигуратор используется разработчиком для создания и изменения конфигурации: справочников, документов, регистров, отчетов, форм и кода. Режим 1С:Предприятие используется пользователями для работы с готовой информационной базой: ввода данных, создания документов и формирования отчетов.
2. Что такое конфигурация и информационная база?
Конфигурация - это описание прикладного решения, то есть набор объектов: справочники, документы, регистры, отчеты, обработки, формы и модули. Она определяет, какие данные хранятся и как с ними работает программа.
Информационная база - это хранилище данных конкретного предприятия, которое работает на основе конфигурации. Конфигурация описывает структуру и логику, а информационная база содержит реальные данные пользователей.

Билет №12. Справочники и реквизиты
1. Что такое справочник?
Справочник - это объект конфигурации 1С, предназначенный для хранения нормативно-справочной информации. Примеры справочников: Товары, Контрагенты, Сотрудники, Склады. Справочник хранит относительно постоянные данные, которые затем используются в документах, отчетах и регистрах.
Основные свойства справочника: код, наименование, реквизиты и возможность иерархии. Код обычно является уникальным идентификатором, наименование - названием элемента, реквизиты - дополнительными полями.
2. Что такое реквизит? Чем отличается от табличной части?
Реквизит - это дополнительное поле объекта, которое хранит одно значение: строку, число, дату, булево значение или ссылку на другой объект. Табличная часть - это вложенная таблица внутри объекта, которая может хранить несколько строк связанных данных.
Например, у справочника Товары реквизитом может быть Цена, а табличная часть могла бы хранить несколько характеристик или комплектующих товара.

Билет №13. Документы и проведение
1. Что такое документ? Чем отличается от справочника?
Документ - это объект конфигурации, который фиксирует хозяйственную операцию во времени. Документ имеет дату и номер. Примеры: Приходная накладная, Реализация товара, Оплата, Заказ покупателя.
Справочник хранит постоянные данные, а документ фиксирует событие. Например, справочник Товары хранит список товаров, а документ Приходная накладная фиксирует факт поступления товаров на склад.
2. Что такое проведение документа?
Проведение документа - это фиксация хозяйственной операции в учетных регистрах. При проведении документ создает движения, которые изменяют остатки, обороты или другие учетные данные. Документ может быть не проведен, проведен или помечен на удаление/отменен.
Кнопка "Записать" сохраняет документ, а "Записать и провести" сохраняет документ и выполняет движения по регистрам.

Билет №14. Регистр накопления и движения
1. Что такое регистр накопления?
Регистр накопления - это объект 1С для хранения изменения числовых показателей во времени. Он используется для учета остатков и оборотов, например остатков товаров на складе или продаж за период.
Регистр накопления может быть вида Остатки или Обороты. Измерения определяют разрез учета, например Товар или Склад. Ресурсы хранят числовые показатели, например Количество или Сумма.
2. Что такое движения документа?
Движения документа - это записи, которые документ создает в регистрах при проведении. Каждое движение может увеличивать или уменьшать ресурс регистра. Для регистра остатков движение имеет вид Приход или Расход.

Регистр сведений - это объект 1С для хранения произвольной информации в разрезе измерений. Он не накапливает остатки и обороты, а хранит значения, например цены товаров, курсы валют, настройки или характеристики.

Билет №18. HTML, CSS и Flexbox
1. Что такое HTML и CSS?
HTML - это язык гипертекстовой разметки, который задает структуру веб-страницы: заголовки, абзацы, ссылки, изображения, формы, таблицы и другие элементы. HTML отвечает за смысл и каркас страницы.

Flexbox - это модель компоновки CSS для расположения элементов в строку или колонку. Она включается свойством display: flex у родительского контейнера. Основные свойства: flex-direction, justify-content, align-items, flex-wrap, gap.

Билет №19. CSS Grid и медиазапросы
1. Что такое CSS Grid? Отличие от Flexbox.
CSS Grid - это двумерная система компоновки. Она работает одновременно со строками и столбцами. Flexbox является одномерным: он располагает элементы либо в строку, либо в колонку. Grid лучше подходит для общей раскладки страницы, а Flexbox - для компонентов.
2. Что такое медиазапросы?
Медиазапросы - это правила CSS, которые применяются при определенных условиях, например при ширине экрана меньше 768px. Они используются для адаптивной верстки, чтобы сайт корректно отображался на компьютере и телефоне.

DOM - это объектная модель документа. Браузер представляет HTML-страницу в виде дерева объектов, где каждый тег является узлом. JavaScript может искать элементы, менять текст, стили, атрибуты, создавать и удалять элементы.

1. Как PHP обрабатывает формы?
PHP получает данные форм через суперглобальные массивы $_GET и $_POST. GET передает данные в URL и подходит для поиска и фильтрации. POST передает данные в теле запроса и применяется для форм, регистрации, логина и сообщений.
2. Защита от XSS
XSS - атака, при которой злоумышленник внедряет JavaScript-код через пользовательский ввод. Для защиты при выводе данных используется htmlspecialchars(), которая превращает специальные символы в HTML-сущности.

Билет №23. PDO и SQL-инъекции
1. Как PHP подключается к MySQL? Что такое PDO?
PDO - универсальный интерфейс PHP для работы с базами данных. Он поддерживает MySQL, PostgreSQL, SQLite и другие СУБД. Подключение выполняется через new PDO(dsn, user, password).
2. Что такое SQL-инъекция?
SQL-инъекция - это атака, при которой вредоносный SQL-код вводится в пользовательское поле и меняет смысл запроса. Защита - подготовленные запросы: prepare() и execute(). Данные передаются отдельно и не воспринимаются как SQL-код.
3. Практика. Каталог товаров через PDO

<?php
try {
    $pdo = new PDO("mysql:host=localhost;dbname=shop;charset=utf8", "root", "", [
        PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION
    ]);
    $stmt = $pdo->prepare("SELECT id, name, price, quantity FROM products");
    $stmt->execute();
    $products = $stmt->fetchAll(PDO::FETCH_ASSOC);

    $totalStmt = $pdo->prepare("SELECT SUM(price * quantity) AS total_sum FROM products");
    $totalStmt->execute();
    $total = $totalStmt->fetch(PDO::FETCH_ASSOC)["total_sum"];
} catch (PDOException $e) {
    die("Ошибка: " . htmlspecialchars($e->getMessage()));
}
?>
